# Task: card-message-protocol

## Goal

Give Agent Bots and incoming webhooks a **standard card message protocol** so
conversations can carry rich, structured, eventually-interactive content
(Feishu/Discord-card-like), renderable consistently on iOS/Android/Web.

This task ships **P1: display-only cards**. A new message payload envelope:

```json
{
  "type": 17,
  "card": { "...": "standard Adaptive Cards JSON, restricted to the octo/v1 profile" },
  "plain": "server-generated authoritative plain text",
  "card_version": "1.5",
  "profile": "octo/v1"
}
```

- `card` is **standard Adaptive Cards 1.5 JSON** restricted server-side to a
  whitelisted subset ("octo/v1 profile"): `TextBlock` (AC basic markdown
  subset — Decision 6), `Image`, `Container`, `ColumnSet`/`Column`, `FactSet`,
  `Action.OpenUrl`, and `selectAction` **only where the carried action is
  `Action.OpenUrl`** (tap-whole-card/region → open link, the Feishu-style
  card-jump form; stateless, hence display-phase-safe — `selectAction`
  inherits the phase of the action it carries). Standard AC syntax (not a
  renamed DSL) so clients may render with AC SDKs or a bespoke renderer
  interchangeably — renderer choice stays reversible without protocol rework.
- `plain` is generated by the server at the dispatch exit (client-supplied
  values are overwritten) via the Decision 8 derivation rule — **never
  empty**. It feeds offline push, search projection, conversation summaries,
  pin tips, copy, downstream LLM consumers, and the degradation chain.
- Interactivity beyond `Action.OpenUrl` (Submit callbacks, inputs, in-place
  card refresh) is **P2, specced in the sibling brief
  `.octospec/tasks/card-message-interaction/`** — published together with this
  one so clients architect against the complete contract in a single pass.
  The phase split is a **contract-freeze order and review boundary, not
  calendar serialization**: P2 implementation may start as soon as both briefs
  are confirmed.

Producer surfaces in P1: `modules/bot_api` (`/v1/bot/sendMessage`, **non-OBO
path only** — see Decision 2b), `modules/robot`, and `modules/incomingwebhook`
native pushes via a new `msg_type: "card"` (alongside existing `"text"` /
`"richtext"`).

### API surface (P1 producers)

Producer wire shapes (authoritative doc: `docs/card-protocol.md`, which
mirrors `pkg/cardmsg/`):

```json
// POST /v1/bot/sendMessage  (existing endpoint, new payload type)
{ "channel_id": "g_9f2c...", "channel_type": 2,
  "payload": { "type": 17, "card": { "...": "..." }, "plain": "ignored — server recomputes",
               "card_version": "1.5", "profile": "octo/v1" } }

// POST /v1/incoming-webhooks/:webhook_id/:token  (existing endpoint, new msg_type)
// "text" is a fallback seed ONLY — used when Decision-8 derivation is empty,
// before the final [卡片] fallback; a text-bearing card body ignores it.
{ "msg_type": "card", "card": { "...": "AC JSON" }, "text": "optional plain seed" }
```

## Responsibility split (职责划分)

Who owns which part of the protocol. This table is normative — implementations
must not move a responsibility across a boundary without a brief revision.

| Party | Owns | Explicitly does NOT own |
|---|---|---|
| **Producers** (`bot_api` / `robot` / `incomingwebhook` callers) | Composing standard octo/v1 AC JSON; fitting the per-ingress body caps | `plain` (server overwrites it); profile enforcement; sender identity |
| **octo-server** (this repo) | **Protocol authority**: envelope schema + octo/v1 whitelist validation (`pkg/cardmsg/`, the single enforcement authority), authoritative `plain` generation at the dispatch exit, size caps, ingress gating (user reject, OBO reject, edit-path reject), offline-push text, search **hit-response** projection, pin/summary display text, i18n error codes. P2 (later): action transport + card update APIs | Rendering; full-text index materialization (indexer's job); client-side degradation |
| **octo-im** | Transparent storage + delivery of opaque payloads; existing CMD sync channel (P2 refresh rides it) | **Zero card awareness — zero code changes in P1.** Deployment check only: WS frame `maxPayloadBytes` (`octo-im pkg/gateway/transport/gnet/ws_frame.go:74`) must be ≥ 1 MiB |
| **Clients** (iOS/Android/Web, dmwork client repos) | Rendering octo/v1 via **bespoke tri-platform renderers (decided — option B**; protocol stays AC-SDK-compatible so the choice is reversible); **the from_uid render gate**; the three-step degradation chain; theming | Accepting cards from non-bot senders; trusting client-supplied `plain` |
| **octo-lib** (follow-up companion PR) | Canonical long-term home for the ContentType constant + card schema (as with `common/richtext.go`) | Blocking P1 — P1 keeps everything in `pkg/cardmsg/` |
| **wukongim-message-indexer** (separate service, follow-up) | Materializing card `plain` into the `searchText` index field (see `modules/messages_search/source.go:9`, `:141-143`) | — full-text *searchability* of cards does not land with octo-server changes alone |

`docs/card-protocol.md` (deliverable of this task) is the client-facing
contract document; it **mirrors** `pkg/cardmsg/` and must say so — the Go
package is the single authority for the whitelist, so the two cannot drift
into competing definitions.

**Renderer decision (maintainer, 2026-07-06): option (B) — bespoke
tri-platform renderers**, implemented in the client repos against the octo
profile + `docs/card-protocol.md`. Consequences: no dependency on the
community AC mobile SDKs, so the former "mobile AC-SDK pilot gate" and "AC
fork ownership" open items are **resolved as moot**; the client teams own
renderer scope (octo/v1 is deliberately small — 6 element types + OpenUrl —
to keep three bespoke implementations cheap and consistent). The evaluated
alternatives stay recorded as references/fallbacks, NOT dependencies:
(A) official AC community mobile SDKs (UIKit/View-based, templating-less,
stagnant mobile investment); (A′) `VikrantSingh01/AdaptiveCards-Mobile`
(MIT, SwiftUI/Compose, AC 1.6 + parity CI — but v0.0.x, bus factor 1, no
known production adoption; useful as an implementation reference for the
bespoke renderers); (C) an octo-profile→DivKit-JSON transpile layer over
DivKit's actively maintained native engines (Apache-2.0; documented Plan B
if bespoke renderer costs blow up). Because the wire payload stays standard
AC, this choice remains reversible — switching to A/A′/C later never reworks
the protocol. The P2 action contract additionally references Slack Block
Kit's `action_id`/`block_id` addressing and Rocket.Chat UiKit's action
lifecycle as production-proven prior art (see sibling brief).

Ecosystem note (source-verified 2026-07-06): third-party agent ecosystems that
emit standard AC (e.g. the OpenClaw `adaptive_card` plugin, same author as A′)
integrate via a **connector** that extracts their card JSON and maps it into
this envelope through the normal bot ingress — zero protocol change. Caveats
recorded from source inspection: that plugin is not published on npm (file:
sibling dependency), its host-adaptation list is hardcoded (no extension
point), and its action model has no idempotency or in-place refresh — our
ingress validation and P2 transport remain the authority for anything it
sends. The one reusable asset in that stack is `adaptive-cards-mcp` (npm
2.3.0, published) as producer-side tooling reference.
